# Real Post Fetching & Display for Cascade Market Discussion

## Context

Cascade has a fully-built discussion UI at `/market/:id/discussion` (in `src/DiscussPage.tsx` and `src/Discussion.tsx`) with threaded posts, stance labels (bull/bear/neutral), post types (argument/evidence/rebuttal/analysis), and reply chains. Today, all discussion content is generated by `generateMockThreads()` (lines 46-296 in DiscussPage.tsx) — fake agents and humans with no persistence.

The **Nostr Integration Setup** plan has established infrastructure:
- **NDK instance** ready (`@nostr-dev-kit/ndk` v3.0.3)
- **Relay configuration** established
- **Event kind & tag structure**: Kind 1 (text notes) with `d` tag (market ID) for scoping
- **Service layer**: `src/services/nostrService.ts` with `fetchEvents()`, `subscribeToEvents()`, `publishEvent()`
- **NostrContext**: `src/context/NostrContext.tsx` providing `useNostr()` hook with pubkey and NDK access

This plan focuses on **removing mock data and fetching real Nostr events for display**. It does NOT cover post creation, reactions, or moderation.

**NOTE**: This plan targets `DiscussPage.tsx`, but post-creation and reactions target `Discussion.tsx`. See Execution Order note about component consolidation.

## Approach

**Recommendation: Hybrid subscription + initial fetch strategy**

Use a **two-phase fetch** model:
1. **Initial Load**: One-time `fetchEvents()` to load existing posts from Nostr (recent kind 1 events with matching market `m` tag)
2. **Live Updates**: Subscribe to new kind 1 events on relevant relays using `subscribeToEvents()` and append them to the thread list in real-time

**Why this approach over alternatives:**
- **Pure fetch-only**: Would require polling to see new posts → wasteful, latent
- **Pure subscription-only**: Users loading the page would miss past posts unless subscription starts beforehand
- **Hybrid (recommended)**: Gives instant historical context + real-time new posts. Matches user expectations from Reddit/Twitter-style discussions

**Tag Structure (from Integration Setup — CORRECTED)**:
```
Kind: 1 (text note)
Query filter for posts in a market:
  {kinds: [1], '#m': [marketId]}

Required tags on posts:
  - m: <market-id>           // market scope (queryable via #m filter)
  - market: <market-id>      // metadata tag for organization
  - stance: bull|bear|neutral // encode position
  - type: argument|evidence|rebuttal|analysis // post category
Reply threading:
  - e: <event-id>            // root event (if reply to root)
  - e: <event-id>            // parent event (if reply to reply)
  - p: <pubkey>              // original author tag
```

### `src/lib/threadBuilder.ts` (Create new)

**Action**: Create

**What**: Helper functions to convert raw Nostr events into the `DiscussionThread` data structure, build reply hierarchies, and handle metadata parsing.

**Why**: Separates complex thread-building logic from the UI component. Handles reply chains (parent/child via e-tags), decodes stance/type tags, and resolves author identities.

**Implementation outline**:

```typescript
import type { NDKEvent } from '@nostr-dev-kit/ndk'
import type { DiscussionThread, Reply } from '../DiscussPage'
import { nostrService } from '../services/nostrService'

/**
 * Convert a single Nostr event into a DiscussionThread root post
 */
export async function convertEventToThread(
  event: NDKEvent,
  allEvents: Map<string, NDKEvent> // for looking up replies
): Promise<DiscussionThread> {
  const tags = nostrService.parseEventTags(event)
  const author = await nostrService.resolveAuthorName(event.pubkey)
  
  return {
    id: event.id || '',
    author: author.name || author.npub.slice(0, 16), // fallback to npub prefix
    isAgent: false, // real posts are not agents (or infer from NIP-05 domain?)
    type: (tags.type || 'argument') as 'argument' | 'evidence' | 'rebuttal' | 'analysis',
    stance: (tags.stance || 'neutral') as 'bull' | 'bear' | 'neutral',
    title: extractTitle(event.content), // first line or first 50 chars
    content: event.content,
    timestamp: (event.created_at || 0) * 1000, // convert unix seconds to ms
    upvotes: 0, // TODO: load from NIP-25 reactions
    downvotes: 0,
    replies: buildReplies(event.id, allEvents, event),
  }
}

/**
 * Build reply chain for a given parent event
 */
function buildReplies(
  parentEventId: string,
  allEvents: Map<string, NDKEvent>,
  parentEvent: NDKEvent
): Reply[] {
  const replies: Reply[] = []
  
  for (const [eventId, event] of allEvents) {
    const tags = nostrService.parseEventTags(event)
    // Check if this event replies to the parent
    if (tags.replyTo === parentEventId || tags.rootId === parentEventId) {
      const author = await nostrService.resolveAuthorName(event.pubkey)
      replies.push({
        id: eventId,
        author: author.name || author.npub.slice(0, 16),
        isAgent: false,
        content: event.content,
        timestamp: (event.created_at || 0) * 1000,
        upvotes: 0, // TODO: load from NIP-25 reactions
        downvotes: 0,
        replies: buildReplies(eventId, allEvents, event), // recursive for nested replies
      })
    }
  }
  
  return replies
}

/**
 * Build the full thread hierarchy from a flat list of events
 * Root events are those with a market 'd' tag but no reply e-tag
 */
export async function buildThreadHierarchy(
  rawEvents: NDKEvent[]
): Promise<DiscussionThread[]> {
  // Map all events by ID for quick lookup
  const allEventsMap = new Map(rawEvents.map(e => [e.id || '', e]))
  
  // Filter to root posts (posts that don't reply to another)
  const rootEvents = rawEvents.filter(event => {
    const tags = nostrService.parseEventTags(event)
    return !tags.replyTo && !tags.rootId
  })
  
  // Convert each root to a DiscussionThread
  const threads = await Promise.all(
    rootEvents.map(event => convertEventToThread(event, allEventsMap))
  )
  
  return threads
}

/**
 * Extract a title from post content
 * Use first line if short, otherwise first 50 chars + ellipsis
 */
function extractTitle(content: string): string {
  const firstLine = content.split('\n')[0]
  if (firstLine.length <= 80) return firstLine
  return firstLine.slice(0, 80) + '...'
}
```

## Subscription vs Fetch Decision

**Recommended: Hybrid (fetch + subscribe)**

| Strategy | Pros | Cons |
|----------|------|------|
| **Fetch only** | Simple, one-time call | Must poll for new posts; stale; user misses live updates |
| **Subscribe only** | Live updates from connection time | User misses past posts unless subscription pre-exists |
| **Hybrid (✓)** | Historical context on load + live updates | Slightly more complex; need to dedupe if event arrives both ways |

The hybrid approach matches Reddit/Twitter behavior: show historical posts on page load, then stream new ones live. For handling deduplication, the `Set<string>` of event IDs can track what's already displayed.
