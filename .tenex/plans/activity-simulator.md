# Cascade Activity Simulator

## Overview

A standalone Node.js script that simulates real activity on the Cascade platform by publishing authentic Nostr events from hundreds of generated keypairs with full kind 0 profiles. Uses Ollama for LLM-generated content (market descriptions, discussion posts, usernames). Runs continuously in the background, creating an organic-feeling stream of activity.

## Architecture

**Single file**: `scripts/simulate.ts` (run via `npx tsx scripts/simulate.ts`)

**Core loop**: Generate a pool of simulated users → bootstrap profiles → continuously create markets, discussions, trades, and reactions at randomized intervals.

**Key principle**: All events are REAL Nostr events published to REAL relays with REAL signatures. The only "fake" thing is that the keypairs are generated for simulation purposes.

## Event Types to Generate

### 1. Kind 0 — Profile Metadata (bootstrap)
- Generate `N` keypairs (configurable, default 200)
- Each gets a kind 0 profile with:
  - `name`: Ollama-generated crypto/prediction-market-themed username
  - `about`: 1-2 sentence bio generated by Ollama
  - `picture`: Use DiceBear API (`https://api.dicebear.com/7.x/identicon/svg?seed=<pubkey>`) — deterministic, no storage needed
  - `nip05`: Optional, `<name>@cascade.market` (won't resolve, but looks real)
- Published once per user at startup, re-published every 24h (replaceable events)

### 2. Kind 982 — Market Events
- Created by random simulated users throughout the day
- Tags: `["d", slug]`, `["title", title]`, `["description", desc]`, `["mint", mintUrl]`, optional `["c", category]`
- Content: Markdown body generated by Ollama (market description + resolution criteria)
- Categories: crypto, politics, tech, sports, science, culture
- Slug format: `<category>-<adjective>-<noun>-<4-digit-hex>` (e.g., `crypto-volatile-ethereum-a3f2`)
- Mint URL: Read from `VITE_CASCADE_MINT_URL` env or default `https://mint.cascade.market`
- Rate: ~1-3 new markets per hour (adjustable)

### 3. Kind 1111 — Discussion Posts (NIP-22)
- Posts on existing markets, referencing the market's kind 982 event ID
- Tags:
  - `["e", marketEventId, "", "root"]` — references the market
  - `["k", "982"]` — kind of root event
  - `["p", marketCreatorPubkey]` — market creator
  - `["stance", "bull"|"bear"|"neutral"]` — user's stance
  - `["type", "argument"|"evidence"|"rebuttal"|"analysis"]` — post type
  - `["subject", title]` — post title
- Content: Ollama-generated discussion text relevant to the market topic
- Replies also supported (kind 1111 with `["e", rootId, "", "root"]` + `["e", parentId, "", "reply"]` + `["k", "1111"]` + `["p", parentAuthorPubkey]`)
- Rate: ~5-15 posts per hour across all markets

### 4. Kind 7 — Reactions (NIP-25)
- Upvotes (`content: "+"`) and downvotes (`content: "-"`) on discussion posts
- Tags: `["e", targetEventId]`, `["p", targetAuthorPubkey]`, `["m", marketId]`
- Rate: ~10-30 reactions per hour

### 5. Kind 30078 — Position Events (NIP-78)
- When a simulated user "trades" on a market, publish a position event
- d-tag: `cascade:position:<marketEventId>:<direction>` (LONG or SHORT)
- Content: JSON with amount, price, timestamp
- Note: In demo mode (no real mint), positions are recorded as kind 30078 events only
- Rate: ~2-5 trades per hour

### 6. Kind 10003 — Bookmarks (NIP-51)
- Some users bookmark markets they're interested in
- Tags: `["e", marketEventId]` for each bookmarked market
- Rate: ~1-2 bookmark updates per hour

## Ollama Integration

### Model Selection
Use `qwen3.5` or `glm-5` (available locally). Fall back to any available cloud model.

### Prompt Patterns

**Username generation** (batch):
```
Generate 50 unique usernames for a prediction market platform. Crypto-native, sharp, opinionated.
Format: one per line, lowercase, no spaces, 5-15 characters.
Examples: satstacker, defioracle, bearwhale, hodl prophet
```

**Market generation**:
```
Create a prediction market about [category]. Title (under 80 chars), description (2-3 sentences), and resolution criteria.
Categories: crypto, politics, tech, sports, science, culture
Output as JSON: {"title": "...", "description": "...", "resolution": "...", "category": "..."}
```

**Discussion post generation**:
```
You are a prediction market user with stance [stance] on this market: "[market title]".
Write a [type] post (2-4 sentences) arguing your position. Be specific, opinionated, reference real events if possible.
```

**Bio generation** (batch):
```
Generate 50 short bios (1-2 sentences) for users on a prediction market platform. Crypto-savvy, opinionated, varying expertise levels.
Format: one per line.
```

### Ollama Client
Use the Ollama HTTP API at `http://localhost:11434/api/generate` — no SDK needed, just `fetch`.

### Batching
- Generate usernames and bios in batches of 50 at startup
- Generate market content on-demand (one at a time, ~30s apart)
- Generate discussion content on-demand (batch 3-5 at a time per market)

## Implementation Details

### Keypair Management
- Use `nostr-tools` (already a dependency) for key generation and event signing
- Generate all keypairs at startup, store in memory as array of `{ privkey, pubkey, profile }`
- Optionally persist to `scripts/.sim-users.json` for reuse across runs (avoid re-publishing profiles every restart)
- If `scripts/.sim-users.json` exists, load from it instead of generating new users

### Event Publishing
- Use `nostr-tools` + WebSocket directly (NOT NDK — simpler for standalone script)
- Connect to relays from the app's testnet config: `wss://relay.nostr.band`, `wss://nos.lol`, `wss://relay.primal.net`, `wss://relay.damus.io`
- Publish to ALL relays for maximum visibility
- Respect relay rate limits: max 1 event per second per relay per user
- Simple retry logic: if publish fails, retry once after 5s, then skip

### Timing & Randomization
- All intervals randomized with jitter to feel organic
- New market: every 20-60 minutes
- New discussion post: every 3-10 minutes
- New reaction: every 1-5 minutes
- New trade: every 10-30 minutes
- Bookmark update: every 30-60 minutes
- Profile re-publish: every 24h
- Use `setTimeout` with randomized delays (not cron) for natural distribution

### State Tracking
- Keep in-memory list of published markets (eventId, title, creator, category)
- Keep in-memory list of published posts (eventId, marketEventId, author, stance, type)
- This allows the script to reference existing events when creating discussions/reactions/replies

### Configuration (env vars)
```
SIM_USER_COUNT=200          # Number of simulated users (default 200)
SIM_MARKET_INTERVAL=30     # Minutes between new markets (default 30)
SIM_POST_INTERVAL=5         # Minutes between new posts (default 5)
SIM_REACTION_INTERVAL=2     # Minutes between reactions (default 2)
SIM_TRADE_INTERVAL=15       # Minutes between trades (default 15)
SIM_MINT_URL=https://mint.cascade.market  # Mint URL for market events
SIM_OLLAMA_MODEL=qwen3.5   # Ollama model to use
SIM_RELAYS=wss://relay.nostr.band,wss://nos.lol,wss://relay.primal.net,wss://relay.damus.io
SIM_PERSIST=scripts/.sim-state.json  # Path to persist state (or empty to disable)
```

### Startup Sequence
1. Load or generate N keypairs
2. Generate profiles via Ollama (batch usernames + bios)
3. Publish kind 0 profiles for all users (rate-limited: 1/sec to avoid relay bans)
4. Seed 3-5 initial markets (Ollama-generated content)
5. Enter main loop with randomized timers

### Main Loop
```
while (true):
  - On market timer: generate + publish new market, announce in log
  - On post timer: pick random market, pick random user, generate + publish discussion post
  - On reaction timer: pick random post, pick random user, publish reaction
  - On trade timer: pick random market, pick random user, publish position event
  - On bookmark timer: pick random market, pick random user, publish bookmark update
  - On profile refresh timer: re-publish a few profiles (kind 0 is replaceable)
```

### Logging
- Simple console output with timestamps
- Format: `[HH:MM:SS] 📊 Market: "Will BTC hit $150k?" by @satstacker`
- Format: `[HH:MM:SS] 💬 Post: "The halving cycle suggests..." by @defioracle (bull on BTC market)`
- Format: `[HH:MM:SS] ⬆️ Reaction: @bearwhale upvoted post by @hodlprophet`
- Format: `[HH:MM:SS] 📈 Trade: @satstacker went LONG on "ETH above $5k" (500 sats)`

### Graceful Shutdown
- On SIGINT/SIGTERM: save state to JSON, close WebSocket connections
- State file allows resuming without re-publishing profiles

## Key Constraints

- **No mock data** — all events are real Nostr events with real signatures
- **Kind 982 only** for markets — not kind 30000
- **E tags only** — never A tags for market references
- **No Nostr jargon** in generated content (per product decisions)
- **Markets never expire** — no expiration mechanism in market events
- **Mint URL** must match the app's configured mint
- **Relay list** must include the app's configured relays
